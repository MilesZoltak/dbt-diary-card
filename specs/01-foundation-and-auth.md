## Overview
This document will outline the HIPAA-compliant foundation of the application. It will detail Firebase Authentication implementation, role-based access control (RBAC) for distinguishing Patients and Clinicians, and the Firestore security rules necessary for encrypting and securing Protected Health Information (PHI) at rest and in transit.

## Implementation Details

### 1. Firebase Authentication & Session Management
- **Methods:** Google Single Sign-On (SSO) as the primary method, providing robust identity verification. Email & Password as a fallback.
- **Multi-Factor Authentication (MFA):** Deprioritized to Phase 2 (P2) given the reliance on Google SSO, which often handles MFA at the provider level.
- **Session Timeout:** Implement an application-level idle timeout of 15 minutes of inactivity that forces a logout or lock screen, ensuring unattended devices do not expose PHI.

### 2. Role-Based Access Control (RBAC)
- **Roles:** `patient`, `clinician`, `system_admin`.
- **Implementation:** Utilize **Firebase Auth Custom Claims** to store user roles (`{ roles: ['clinician', 'patient'] }`). A user can hold multiple roles, allowing clinicians to test the app as a patient or be patients themselves.
- **Onboarding Flow:**
  - *Clinicians* are manually provisioned by a `system_admin`.
  - *Patients* register via an invitation link generated by their Clinician, which securely links their `uid` to the Clinician's roster upon account creation.

### 3. Data Architecture & PHI Isolation (Unified Model)
To support users acting as both clinicians and patients, we will use a unified `users` collection with role-specific subcollections:
- **Collection Structure:**
  - `/users/{uid}`: Core user profile (name, email, role flags).
  - `/users/{uid}/diaryCards/{cardId}`: The actual PHI payload for diary entries. *Every* user has this, allowing clinicians to track their own data or test the patient experience.
  - `/users/{uid}/patientData/{docId}`: Patient-specific configurations and PHI metadata.
  - `/users/{uid}/clinicianData/{docId}`: Clinician-specific configurations, templates, and settings.
  - `/users/{uid}/roster/{patientId}`: For clinicians, a mapping granting them explicit access to specific patients they treat.

### 4. Firestore Security Rules (HIPAA Enforcement)
- **Default Policy:** `allow read, write: if false;` (Deny all).
- **Self Access:** `allow read, write: if request.auth.uid == uid;` Users can always access their own data in `/users/{uid}/*`.
- **Clinician Access:** Clinicians can only read a patient's data (`/users/{patientId}/diaryCards/*`) if the patient's ID exists in the clinician's `/users/{uid}/roster/` subcollection. Clinicians *cannot* modify patient self-reported data (read-only access).
- **Emergency Access:** Standard RBAC is deemed sufficient; no "break-glass" emergency protocol is required.
- **Validation:** Rules will enforce schema validation (checking for unexpected fields) to prevent malformed data injection.

### 5. Encryption & Audit Logging
- **Encryption at Rest/Transit:** Rely on Google Cloud's default AES-256 encryption at rest and TLS 1.2+ for data in transit. (Note: A Business Associate Agreement (BAA) with Google should be signed).
- **Audit Logging:** 
  - *Writes/Updates:* Tracked via metadata fields (`createdAt`, `updatedAt`, `modifiedBy`) on documents.
  - *Reads of PHI:* We will enable **GCP Data Access Audit Logs** for Firestore to track who viewed what PHI. This natively tracks all reads without custom code. *(Cost context: The first 50 GiB of log ingestion per month is free in GCP, which is highly sufficient for this scale. Beyond that, it is $0.50/GiB).*
